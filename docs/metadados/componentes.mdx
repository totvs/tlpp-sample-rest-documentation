# Metadado `TLPP COMPONENT`

Schemas **reutilizáveis** para `requestBody` e `responses` no JSON DOC. No OpenAPI gerado viram entradas em `components/schemas` referenciadas por `$ref`.

Se você viu `"type": "component"` no JSON da função `_DOC` e não soube o que era — é isso: um atalho para não repetir as mesmas propriedades em cada endpoint.

## Por que usar

| Sem component | Com component |
|-------------|---------------|
| Copia `param1`, `sucess`, `error`… em cada `_DOC` | Declara uma vez em `TLPP COMPONENT` |
| OpenAPI com schemas duplicados | Um schema central + `$ref` em vários paths |

## Passo 1 — Declarar no fonte

Arquivo `.tlpp` separado, **compilado no RPO** (ex.: `src/components/sample_components.tlpp`):

```tlpp
#include "tlpp-doc.th"

/*
  TLPP COMPONENT <nome> <propriedade> <tipo AdvPL> <exemplo>
  Mesmo <nome> em várias linhas = um objeto com várias propriedades.
*/

TLPP COMPONENT sampleBodyIn param1 character "tlpp"

TLPP COMPONENT sampleBodyOutOk sucess logical "true"
TLPP COMPONENT sampleBodyOutOk error  character ""

TLPP COMPONENT sampleBodyOutError sucess logical "false"
TLPP COMPONENT sampleBodyOutError error  character "Invalid Path parameter"
```

| Parte | Significado |
|-------|-------------|
| `sampleBodyIn` | Nome do componente — usado no JSON DOC |
| `param1` | Propriedade do objeto |
| `character` | Tipo AdvPL (`logical`, `integer`, `numeric`, `date`, …) |
| `"tlpp"` | Exemplo no OpenAPI |

## Passo 2 — Referenciar no JSON DOC

Na função `_DOC`, dentro de `requestBody` ou de cada item em `responses`, informe o Content-Type e aponte para o componente:

### `requestBody`

```tlpp
jBodyTypeJson := JsonObject():New()
jBodyTypeJson['json'] := JsonObject():New()
jBodyTypeJson['json']['type'] := "component"
jBodyTypeJson['json']['component'] := "sampleBodyIn"

jReqBody := JsonObject():New()
jReqBody['description'] := "Body param"
jReqBody['Content-Type'] := { jBodyTypeJson }

jDoc['requestBody'] := jReqBody
```

A chave do content-type pode ser `json`, `application/json`, `application/xml`, etc. — o nome deve bater com o que você documenta para aquele endpoint.

### `responses`

```tlpp
jRes200TypeJson := JsonObject():New()
jRes200TypeJson['application/json'] := JsonObject():New()
jRes200TypeJson['application/json']['type'] := "component"
jRes200TypeJson['application/json']['component'] := "sampleBodyOutOk"

jResp200 := JsonObject():New()
jResp200['statusCode'] := 200
jResp200['description'] := "Sucesso"
jResp200['Content-Type'] := { jRes200TypeJson }

jRes400TypeJson := JsonObject():New()
jRes400TypeJson['json'] := JsonObject():New()
jRes400TypeJson['json']['type'] := "component"
jRes400TypeJson['json']['component'] := "sampleBodyOutError"

jResp400 := JsonObject():New()
jResp400['statusCode'] := 400
jResp400['description'] := "Requisição inválida"
jResp400['Content-Type'] := { jRes400TypeJson }

jDoc['responses'] := { jResp200, jResp400 }
```

Cada status pode usar um componente diferente (`sampleBodyOutOk` no 200, `sampleBodyOutError` no 400).

## Estrutura resumida do JSON

```json
{
  "requestBody": {
    "description": "...",
    "Content-Type": [
      {
        "json": {
          "type": "component",
          "component": "sampleBodyIn"
        }
      }
    ]
  },
  "responses": [
    {
      "statusCode": 200,
      "Content-Type": [
        {
          "application/json": {
            "type": "component",
            "component": "sampleBodyOutOk"
          }
        }
      ]
    }
  ]
}
```

Monte com `JsonObject():New()` — o exemplo acima é só para leitura.

## O que sai no OpenAPI

O motor gera `$ref` para `components/schemas`:

```yaml
requestBody:
  content:
    application/json:
      schema:
        $ref: '#/components/schemas/sampleBodyIn'

components:
  schemas:
    sampleBodyIn:
      type: object
      properties:
        param1:
          type: string
          example: tlpp
    sampleBodyOutOk:
      type: object
      properties:
        sucess:
          type: boolean
          example: true
        error:
          type: string
```

Veja o arquivo completo em [OpenAPI gerado](/docs/tlpp/rest/doc-generate/openapi) (`api_doc_8080.yaml`).

## `component` vs schema inline

| | `"type": "component"` | Propriedades inline no `JsonObject` |
|---|----------------------|-------------------------------------|
| Declaração | `TLPP COMPONENT` no `.tlpp` | Campos `properties` dentro da `_DOC` |
| Reuso | Mesmo nome em N endpoints | Copiar em cada função |
| OpenAPI | `components/schemas` + `$ref` | Schema embutido no path |

Prefira **component** quando o mesmo body aparece em mais de um endpoint ou quando request e response compartilham estrutura parecida.

## `tlpp.doc.getComponent()`

Em `tlpp.doc.generate('swagger', ...)`, o motor resolve componentes automaticamente no YAML.

Se você lê a documentação em **formato JSON** (não Swagger) e o payload traz só o **nome** do componente, use `tlpp.doc.getComponent()` para expandir o schema com propriedades e exemplos.

## Checklist

1. `#include "tlpp-doc.th"` no arquivo de componentes
2. Fonte compilado no mesmo RPO das APIs
3. No `_DOC`: `"type": "component"` **e** `"component": "nomeExato"`
4. Nome case-sensitive — igual ao `TLPP COMPONENT`
