# JSON retornado pela função DOC

Quando `description="[NomeFunc_DOC]"`, o motor executa a função e espera um **JSON** com os campos OpenAPI do endpoint.

## Como montar o JSON

**Use `JsonObject():New()` e `:toJson()`** — não concatene strings manualmente. Objetos nativos garantem JSON válido.

```tlpp
local jDoc := JsonObject():New()
jDoc['title'] := "DOC simples 04"
jDoc['params'] := { jParamQuery, jPathParam }
return jDoc:toJson()
```

## Campos suportados

| Campo | Função |
|-------|--------|
| `title` | Título do endpoint |
| `description` | Texto descritivo |
| `description1`…`N` | Descrição longa em blocos numerados |
| `params` | Array de parâmetros (`name`, `in`, `type`, `required`, …) |
| `requestBody` | Corpo da requisição (`Content-Type`, schema inline ou [component](/docs/tlpp/rest/metadados/componentes)) |
| `responses` | Array de respostas (`statusCode`, `description`, schemas ou `component`) |

## Referência a component (`"type": "component"`)

Para bodies reutilizáveis, não descreva cada propriedade na `_DOC`. Aponte para um `TLPP COMPONENT` declarado em outro `.tlpp`:

```tlpp
jBody['json']['type'] := "component"
jBody['json']['component'] := "sampleBodyIn"
```

O motor expande para `components/schemas` no OpenAPI. Declaração do componente, exemplos de request/response e o YAML resultante: [Componentes](/docs/tlpp/rest/metadados/componentes).

## Multi-linha — regra crítica

Ao usar `description1`, `description2`, `description3`, … **não salte números**. Se pular do `description3` para o `description5`, o motor interrompe silenciosamente no gap.

## `translate:` no JSON

Chaves no formato `translate:Funcao:ID:texto` são resolvidas pelo dicionário i18n — ver [i18n](/docs/tlpp/rest/metadados/i18n).
